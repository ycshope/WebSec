# 任务27：AD身份认证；AD横向移动；AD持久化；

## AD的身份认证

AD支持多种认证协议，支持Windows、Linux、MacOS系统集中认证，大多数情况AD使用Kerberos、NTLM两种认证方式。

### NTML认证过程

NTLM authentication is a simpler process compared to Kerberos and is less secure. Here's a detailed breakdown of the NTLM authentication process:

1. **User Authentication Request:**
   - The user logs in to their client machine and attempts to access a network resource.
2. **Challenge-Response Mechanism:**
   - The resource server challenges the client to prove its identity.
   - The challenge is a random value generated by the resource server.
3. **Hashed Password:**
   - The client hashes the user's password and sends the hash to the resource server as a response to the challenge.
4. **Comparison:**
   - The resource server retrieves the stored password hash from the Active Directory database for the user's account.
   - It compares the received hash with the stored hash.
   - If the hashes match, the user is granted access to the resource.

### Kerberos认证过程

Kerberos authentication involves multiple steps to establish a secure connection between a client and a network resource. Here's a detailed breakdown of the Kerberos authentication process:

1. **User Authentication Request:**
   - The user logs in to their client machine and attempts to access a network resource, such as a file share or a web application.
2. **Authentication Ticket Request (TGT):**
   - The client requests an Authentication Ticket (TGT) from the Authentication Server (AS)(这个AS通常是域控.
   - The request includes the user's identity and is encrypted with a secret key derived from the user's password.
3. **Authentication Server Verification:**
   - The AS verifies the user's credentials by checking the username and password against the Active Directory database.
   - If the credentials are valid, the AS creates a TGT encrypted with a secret key shared with the Ticket Granting Server (TGS).
4. **TGT Response:**
   - The AS sends the TGT back to the client.
5. **Service Ticket Request (TGS Request):**
   - When the client needs access to a specific network resource, it requests a service ticket from the Ticket Granting Server (TGS).
   - The request includes the TGT and the name of the target resource.
6. **Service Ticket Generation:**
   - The TGS verifies the TGT and the user's identity.
   - If everything checks out, the TGS generates a service ticket for the requested resource.
   - This service ticket is encrypted with a session key that will be used for secure communication with the resource server.
7. **Accessing the Resource:**
   - The client presents the service ticket to the resource server.
   - The resource server decrypts the service ticket using its own secret key.
   - It verifies the client's identity and checks if the service ticket is valid and hasn't expired.
   - If everything is in order, the resource server grants access to the requested resource.
8. **Secure Session:**
   - Once the resource server and the client have verified each other's authenticity, they establish a secure session using the session key contained in the service ticket.
   - This session key is used for encrypting and decrypting data exchanged during the session.

Kerberos provides strong security features, including mutual authentication, ticket expiration, and secure key exchange. It helps protect against various forms of attacks, such as eavesdropping and replay attacks.

## 攻击模式

### **密码爆破**

通过mimikatz读取域控密码哈希(如果该设备用域控登录过)，再进行字典破解，该攻击仅限于弱口令



### **Pass the Hash (PtH):**

**What is Pass the Hash (PtH)?**

 Pass the Hash is a type of cyberattack where an attacker steals the hashed password value from one system and uses it to authenticate and gain unauthorized access to other systems or resources without needing to know the actual plaintext password. This attack is most commonly associated with NTLM authentication.

**How Does Pass the Hash Work?** 

Here's a detailed breakdown of the Pass the Hash attack:

1. **Hash Extraction:** The attacker gains access to a system where password hashes are stored or intercepted. This can be done through various means, such as exploiting vulnerabilities, using malware, or capturing network traffic.
2. **Hash Usage:** Instead of attempting to crack the hashed password (which could be time-consuming), the attacker simply uses the stolen hash value to authenticate to other systems.
3. **Passing the Hash:** The attacker sends the stolen hash to the target system, which attempts to verify the hash against its own stored hash. If the hashes match, the attacker gains access without needing to know the actual password.
4. **Unauthorized Access:** With access to the target system, the attacker can escalate privileges, move laterally within the network, and potentially compromise other systems.



### **Pass the Ticket (PtT):**

**What is Pass the Ticket (PtT)?**

 Pass the Ticket is a cyberattack technique associated with the Kerberos authentication protocol. In this attack, an adversary steals a valid Kerberos ticket-granting ticket (TGT) and uses it to impersonate a legitimate user or service to gain unauthorized access to resources.

**How Does Pass the Ticket Work?** 

Here's a detailed breakdown of the Pass the Ticket attack:

1. **Ticket Theft:** The attacker gains access to a user's or service's TGT. This can be achieved through various means, such as compromising a user's credentials, using malware, or exploiting vulnerabilities.
2. **Ticket Usage:** The attacker uses the stolen TGT to request service tickets from the Ticket Granting Server (TGS) without the need to know the user's password.
3. **Impersonation:** With the service ticket in hand, the attacker presents it to a target service as if they were the legitimate user or service, gaining unauthorized access.
4. **Unauthorized Access:** Once authenticated, the attacker can access resources and potentially perform malicious actions on behalf of the compromised user or service.



### **Golden Ticket**

A Golden Ticket is an unauthorized Kerberos ticket-granting ticket (TGT) that has been forged or created by an attacker who has compromised the Kerberos Ticket Granting Server (TGS) key. It is one of the most powerful and dangerous forms of attack because it essentially allows the attacker to impersonate any user or service in the domain without needing to know their password.

Here's how a Golden Ticket attack works:

1. **Initial Compromise:** The attacker gains administrative access to a domain controller and extracts the Kerberos TGS key, which is used to encrypt TGTs.
2. **TGT Generation:** With the TGS key, the attacker can create a TGT for any user or service in the domain, effectively granting them full access to the network.
3. **Unauthorized Access:** The attacker presents the forged TGT to any service within the domain, and because it is cryptographically valid, the service grants access, believing the attacker is the legitimate user or service.
4. **Complete Control:** The attacker can move laterally through the network, access sensitive resources, and potentially escalate privileges, all while remaining undetected because they have effectively become a "golden" user with unrestricted access.



### **Silver Ticket**

A Silver Ticket is another type of unauthorized Kerberos ticket, but unlike a Golden Ticket, it doesn't grant full domain-level access. Instead, it allows an attacker to impersonate a specific service or resource within the domain. Silver Tickets are typically used for more targeted attacks.

Here's how a Silver Ticket attack works:

1. **Initial Compromise:** The attacker gains access to a user's or service account's credentials or acquires a user's TGT through various means.
2. **Service Ticket Generation:** Using the compromised credentials or TGT, the attacker forges a service ticket for a specific service or resource within the domain.
3. **Unauthorized Access:** The attacker presents the forged service ticket to the targeted service. If the ticket is valid, the service grants access, believing the attacker is the legitimate user or service for which the ticket was generated.
4. **Limited Access:** The attacker has access to only the specific resource or service associated with the Silver Ticket, rather than complete domain access.

Silver Ticket attacks can be used for lateral movement within the network or to target specific resources of interest. To mitigate Silver Ticket attacks, organizations should implement strong access controls, monitor for suspicious activity, and follow best practices for credential management.
